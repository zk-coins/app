zkCoins App — Handbook

User-Doku und Test-Doku in einem. Jedes Bild auf dieser Seite ist gleichzeitig eine Visual-Regression-Baseline, die in CI bei jedem PR pixelgenau verglichen wird. Wenn das Handbook aktuell aussieht, ist die Live-App es auch — und umgekehrt.

🟢 PRD: zkcoins.app 🟡 DEV: dev.zkcoins.app 📂 Code: zk-coins/app 🧪 Specs: 11 · 70 Tests · 67 Baselines

Single Source of Truth

Die kanonischen Baseline-Bilder liegen unter e2e/<spec>.spec.ts-snapshots/<name>-chromium-linux.png. Sie sind das einzige committed Asset und werden ausschließlich über den Regenerate Visual Baselines-Workflow erneuert. Die Bilder, die dieses Handbook zeigt, sind Hardlinks (scripts/sync-handbook-baselines.mjs) — kein zweites Original, kein Drift möglich.

Jeder Test in jedem Spec-File entspricht genau einem Screen (oder einer Screen-Variante), den die App rendern kann. Wenn ein Test grün ist, ist die Variante visuell wie spezifiziert. Wenn ein Test rot wird, ist entweder die UI gebrochen oder absichtlich verändert — letzteres erfordert einen Baseline-Regen.

DEV spiegelt PRD. Beide Deployments bauen mit denselben NEXT_PUBLIC_ENABLE_*-Defaults (alle aus). Dead-Code-Elimination strippt jeden gated Pfad identisch. Was du in diesem Handbuch siehst, ist das, was auch live ist — kein Feature im Screenshot versteckt sich in Produktion hinter einem Flag. Feature-Flags gibt es nur lokal via .env.local.

Viewports. Screenshots zeigen standardmäßig die Mobile-Ansicht 375 × 812 — zkCoins ist eine Mobile-First-PWA. Paired Desktop-Varianten 1440 × 900 sind für die fünf Bildschirme erfasst, deren Layout auf breiteren Viewports strukturell anders aussieht (Welcome, Settings, Funded Balance, Send-Confirm-Dialog, Receive). Welcome hat zusätzlich eine Tablet-Baseline 768 × 1024 am md:-Breakpoint. Test-spezifische Hover-Zustände bleiben Desktop, weil :hover eine Desktop-only-Interaktion ist.

04Wallet entsperren — Passwort

e2e/04-unlock-password.spec.ts
4 Tests · 4 Baselines
Spec ↗ Baselines ↗

Bei jedem App-Start mit bereits liegender verschlüsselter Wallet zeigt die App den Unlock-Screen. PBKDF2 leitet aus dem Passwort einen AES-Schlüssel ab; AES-GCM wirft bei falschem Passwort einen Authentication-Tag-Mismatch, was die App als „Wrong password" anzeigt. Schließt das frühere MVP-Triage-Gap. MVP Gap closed

„Welcome back"-Hero, leeres Passwort-Feld, „Unlock"-Button disabled.
unlock-typedL50
Passwort eingegeben — „Unlock" wird aktiv. Kein Validation auf Länge etc.; die Härte des Passworts ist die ganze Sicherheit.
unlock-wrong-errorL74
AES-GCM-Tag-Mismatch → rote Fehlermeldung direkt unter dem Feld. Kein Rate-Limit, kein Cooldown.
unlock-success-walletL84
Erfolgreich entsperrt — Wallet öffnet sich mit aktuellem Stand.

Baselines-Regeneration

Wenn UI absichtlich verändert wird, regeneriert man die Baselines via gh workflow run "Regenerate Visual Baselines" --ref develop -f branch=develop. Der Workflow öffnet einen Side-Branch-PR mit den neuen PNGs, der per gh pr merge --admin --squash gemergt wird. Niemals Baseline-PNGs manuell editieren. Vollständiger Plan in e2e/README.md.

Triage-Gaps (explizit nicht abgedeckt)

  • Passkey-Flows (gated) — funktional via webauthn.spec.ts getestet, keine Visual-Baseline.
  • Network Activity Chart (keep) — simulierte Daten brechen Pixel-Diff, kein dedizierter Test.
  • Nativer Browser-Install-Dialog — nicht headless exerzierbar.
  • Dev-Routes (/reset, /simulate) — Dev-only.
