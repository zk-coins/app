zkCoins App — Handbook

User docs and test docs in one. Every image on this page is also a visual-regression baseline, pixel-compared in CI on every PR. If the handbook looks current, the live app is too — and the other way around.

🟢 PRD: zkcoins.app 🟡 DEV: dev.zkcoins.app 📂 Code: zk-coins/app 🧪 Specs: 11 · 73 tests · 70 baselines

Single source of truth

The canonical baseline images live under e2e/<spec>.spec.ts-snapshots/<name>-chromium-linux.png. They are the only committed asset and can only be refreshed via the Regenerate Visual Baselines workflow. The images this handbook displays are hardlinks placed by scripts/sync-handbook-baselines.mjs — no second copy, no possible drift.

Every test in every spec file maps to exactly one screen (or screen variant) the app can render. If a test is green, that variant looks as specified. If a test goes red, either the UI broke or it was intentionally changed — the latter requires a baseline regen.

DEV vs. PRD bundle: the suite runs against dev.zkcoins.app (DEV) because that's the only deployment where the faucet (POST /api/mint) is available. The DEV bundle additionally shows the Apps tab in the bottom nav and the faucet button in the wallet's empty state — both are build-time flags stripped from the PRD bundle by dead-code elimination. The 10 MVP functions are identical in both bundles.

04Unlock wallet — Password

e2e/04-unlock-password.spec.ts
5 tests · 5 baselines
Spec ↗ Baselines ↗

On every app start with an encrypted wallet already on the device, the app shows the unlock screen. PBKDF2 derives an AES key from the password; AES-GCM throws an authentication-tag mismatch on a wrong password, which the app surfaces as "Wrong password". Closes the former MVP triage gap. MVP Gap closed

unlock-emptyL41
"Welcome back" hero, empty password field, "Unlock" button disabled.
unlock-typedL50
Password entered — "Unlock" becomes active. No length validation; the hardness of the password is the entire security.
unlock-unlockingL59
Loading state while PBKDF2 runs (200–400 ms depending on device). The delay is intentional and makes brute force expensive.
unlock-wrong-errorL74
AES-GCM tag mismatch → red error directly under the field. No rate limit, no cooldown.
unlock-success-walletL84
Successfully unlocked — wallet opens with current state.

Required CI checks on develop

  • Lint & Build

Baseline regeneration

When the UI is intentionally changed, regenerate baselines via gh workflow run "Regenerate Visual Baselines" --ref develop -f branch=develop. The workflow opens a side-branch PR with the new PNGs, merged via gh pr merge --admin --squash . Never edit baseline PNGs manually. Full plan in e2e/README.md.

Triage gaps (explicitly not covered)

  • Passkey flows (gated) — covered functionally via webauthn.spec.ts, no visual baseline.
  • Network activity chart (keep) — simulated data breaks pixel diff, no dedicated test.
  • Native browser install dialog — not headless-exercisable.
  • Dev routes (/reset, /simulate) — DEV-only.
